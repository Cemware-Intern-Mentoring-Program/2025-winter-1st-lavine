> 주제 2: Spring Framework의 주요 특징은 무엇인가? 그리고 그 특징이 어떻게 구현될 수 있었을까?

## 스프링의 핵심 해결책: IoC, DI, AOP

### IoC/DI

스프링이 EJB의 복잡성을 해결하고 경량화를 이룬 핵심 원리는 **'제어의 역전(IoC)'** 과 이를 구현하는 **'의존성 주입(DI)'** 이다. 

| 개념 | 설명 | EJB 대비 장점 |
| :--- | :--- | :--- |
| **제어의 역전 (IoC, Inversion of Control)** | **객체의 생성, 관리, 라이프사이클**에 대한 **제어권**이 개발자 코드(`new` 키워드)에서 프레임워크(스프링 컨테이너)로 넘어가는 것. | 개발자가 객체 생성 대신 **순수 비즈니스 로직**에만 집중할 수 있게 되어 코드가 단순해진다. |
| **의존성 주입 (DI, Dependency Injection)** | 객체가 필요로 하는 다른 객체(의존성)를 개발자가 직접 만들지 않고, 컨테이너가 자동으로 연결(주입)해 주는 IoC의 구체적인 구현 방식. | EJB처럼 복잡한 설정 없이, 객체들을 유연하게 조립(Coupling)하여 사용할 수 있으며 테스트 용이성이 극대화된다. |

#### 예시 코드
적용 전
```Java
class Engine {
    public void start() {
        System.out.println("Engine started.");
    }
}

class Car {
    private Engine engine; 

    public Car() {
        // Car가 Engine 객체를 직접 생성하고 제어
        this.engine = new Engine(); 
    }

    public void drive() {
        engine.start();
        System.out.println("Car is driving.");
    }
}

public class NonIocExample {
    public static void main(String[] args) {
        Car myCar = new Car(); 
        myCar.drive();
    }
}
```
적용 후
```Java
class Engine {
    public void start() {
        System.out.println("Engine started.");
    }
}

class Car {
    private Engine engine; 

    // 생성자를 통해 외부로부터 Engine 객체를 주입받음
    public Car(Engine engine) {
        // Car는 Engine을 직접 생성하지 않고, 받은 것을 사용하기만
        this.engine = engine; 
    }

    public void drive() {
        engine.start();
        System.out.println("Car is driving.");
    }
}

public class IocDiExample {
    public static void main(String[] args) {
        Engine newEngine = new Engine(); 
        
        Car myCar = new Car(newEngine); 

        // Car는 Engine이 어떻게 만들어졌는지 알 필요 없이 운전만 
        myCar.drive();
    }
}
```


### AOP
IoC와 DI를 통해 객체 간의 의존 관계를 느슨하게 만들어주었다면, AOP는 애플리케이션 전반에서 반복적으로 나타나는 기능(횡단 관심사)을 모듈화하여 핵심 로직과 분리해주는 개념이다. 

| 개념                    | 설명                                                  | IoC/DI & EJB 대비 장점                          |
| :-------------------- | :--------------------------------------------------------------- | :--------------------------------------------------- |
| **AOP (관점 지향 프로그래밍)** | 로깅/보안/트랜잭션과 같이 여러 모듈에 **공통적으로 사용되지만 핵심 로직이 아닌 기능**을 분리하여 관리하는 방식 | 중복이 제거되어 핵심 비즈니스 로직이 깔끔해지고, 공통 기능을 변경해도 한 곳만 수정하면 된다 |

로깅이나 트랜잭션 처리, 로깅, 인증/인가 체크 등은 대부분의 서비스에서 계속 반복된다. 이를 코드마다 삽입하게 되면 중복이 생기고 가독성이 떨어진다 -> 유지보수 난이도 증가

이런 상황에서, 비즈니스 코드를 핵심 로직만 남기고 그 외 부가적인 기능을 독립적으로 분리하여 관리하도록 하는 방식을 AOP라고 한다.
