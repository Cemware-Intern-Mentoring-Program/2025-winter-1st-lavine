## 서버와 클라이언트는 무엇인가
서버와 클라이언트는 물리적 관점에서 둘 다 하나의 컴퓨터(하드웨어)로 볼 수 있다. 이 때 컴퓨터 위에서 동작하는 소프트웨어의 역할에 따라 서버와 클라이언트 역할이 나뉘며, 이 역할은 고정된 것이 아니라 상대적이다.

- 서버 : 
    - 정의 : 네트워크를 통해 클라이언트의 요청을 받아 처리하고 그에 대한 응답을 제공하는 주체이다.
    - 역할 : 데이터 저장, 관리, 계산 등 서비스 기능을 수행 가능
    - 예시 : 웹 페이지 파일을 제공하는 웹 서버, 데이터베이스를 관리하는 DB서버 등이 있다.
- 클라이언트 : 
    - 정의 : 서버에 필요헌 정보를 요청하고, 서버로부터 응답을 받아서 이용하는 주체
    - 역할 : 사용자와의 상호작용을 담당하고, 받은 데이터를 표현하는 역할을 한다. 
    - 예시 : 웹 브라우저, 스마트폰의 앱, 데스크톱 프로그램 등이 있다.

## 자바 웹 프레임워크의 역사

자바 웹 프레임워크의 역사를 기술 등장 시점과 기존 기술의 한계를 극복하고자 하는 시간 흐름에 따라 정리하면 다음과 같이 정리할 수 있다. 

> 서블릿 -> JSP -> MVC 패턴 -> 스트럿츠, 웹워크 -> 스프링 MVC

### 1. 서블릿 (Servlet)
초창기 web은 정적 데이터만 전달하는 Web Server였다.

즉, 사용자의 요청에 따라 다양한 동적 처리를 하는 것이 불가능했다.

#### 특징
- 서블릿은 자바를 이용해 동적인 웹 콘텐츠를 생성하기 위해 등장한 최초의 표준 기술이다. 
- 서버 측에서 실행되며, 클라이언트의 HTTP 요청을 받아 처리하고 응답을 생성하는 자바 클래스이다.

#### 단점
초기 서블릿은 TCP/IP 연결이나 멀티 스레드 처리 등의 작업을 직접 구현해야 하는 등의 단점이 존재하고 여전히 동적으로 HTML을 생성하기 어렵다는 문제가 있었다.


### 2. JSP (JavaServer Pages)
서블릿의 HTML 작성 어려움을 해소하기 위해 등장하였다. 

#### 특징
- HTML을 포함하면서도 자바 코드 작성이 가능하다
    - HTML 구조를 기반으로ㅡ 필요한 곳에 `<% ... %>`같은 태그를 사용해 자바 코드를 삽입하는 방식
    - 개발자가 UI 구현에 집중할 수 있게 되면서 생산성 향상 효과
- JSP 코드는 최종적으로 서블릿으로 변환된다. 
    - 웹 컨테이너(WAS)에 의해 자바 파일(.java)로 변환되고, 다시 클래스(.class)로 컴파일되어 서블릿으로 동작한다.


#### 단점
JSP의 편의성으로, 개발자들이 데이터 처리 로직까지 JSP 안에 작성하기 시작하면서 유지보수가 어렵다는 문제가 발생하였다.
이로 인해 하나의 JSP파일에 DB연결, 계산 로직, 화면 표시 등의 코드가 섞이면서 이해하고 수정하기 어려워지는 문제가 발생

### 3. MVC 패턴 도입
MVC 패턴은 JSP의 가장 큰 문제점이었던 비즈니스 로직과 화면 로직 혼재 해결을 위해 고안된 디자인 패턴이다.

> MVC : 소프트웨어의 역할을 세가지 요소로 나누어 분리하여, 서로 독립적으로 변경하거나 확장할 수 있도록 설계한다. 
> - Model : 비즈니스 로직 및 데이터 관리 담당 (순수 Java 객체)
> - View : 사용자에게 데이터를 표시하는 화면 담당 (ex. JSP)
> - Controller : 요청을 받고 Model과 View를 연결하는 중간 제어 역할 담당 (ex. Servlet)

이 패턴을 통해 JSP는 순수하게 화면 역할만 담당하게 되어 코드가 단순해지고, 개발자는 로직과 화면을 분리하여 효율적인 유지 보수가 가능해졌다.

장점은 다음과 같다. 
- 동시다발적 개발 가능 : 프론트와 백이 독립적 개발진행
- 높은 응집도: 논리적으로 관련있는 기능을 하나의 컨트롤러로 묶거나, 특정 모델과 관련 있는 뷰로 그룹화
- 낮은 의존도: 뷰, 모델, 컨트롤러 각각 독립적이다
- 개발용이성: 책임이 구분되어 있어 코드 수정 편안
- 한 모델에 대한 여러 형태의 뷰를 가짐: HTML, JSON …

단점으로는 배우기 어렵고 쓰기 어렵다는 점이 있다.

### 4. J2EE(Java 2 Platform, Enterprise Edition)
MVC 패턴이 로직과 화면 분리를 통해 구조적 문제를 해결했다면 J2EE는 대규모 기업 환경에서 발생하는 복잡한 요구사항(트랜잭션 관리, 분산 환경 처리, 보안, 원격 통신 등)들을 표준화하기 위해 등장했다.

> J2EE : 서블릿, JSP를 포함하여 EJB 등 기업용 애플리케이션 개발에 필요한 다양한 기술 스펙을 총칭하는 자바의 기업용 플랫폼 표준

J2EE의 등장으로 개발자가 **복잡한 인프라 기술이나 비즈니스 외적인 문제에 신경쓰지 않고 순수하게 비즈니스 로직 개발에 집중할 수 있는 환경**이 되었다.

### 5. EJB (Enterprise JavaBeans)

> J2EE 표준의 핵심적인 기술 컴포넌트로, 대규모 시스템의 복잡한 요구사항을 컨테이너가 자동으로 처리해주는 것을 목표로 했다.

다음과 같은 특징을 가진다.
- 컴포넌트 모델 : 비즈니스 로직을 담는 일종의 자바 컴포넌트(빈)이다.
- 컨테이너 위임 : EJB를 실행하는 EJB 컨테이너가 트랜잭션, 보안, 원격 호출 등의 복잡한 기술적 기능을 개발자 대신 처리해준다. 

하지만 다음과 같은 치명적 단점도 존재한다.
- 과도한 복잡성
     - 단순한 기능을 구현하려 해도 EJB 스펙을 맞추기 위해 수많은 인터페이스와 클래스를 만들어야 했고, 방대한 양의 XML  설정 파일 작성이 필요했다.
    - 구조 자체가 복잡하여 배우고 적용하는데 시간이 오래걸린다. 
- 무거운 기술
    - EJB 컨테이너를 포함하는 WAS 자체가 매우 무겁고 느렸다. 개발 및 테스트 과정에서 매번 서버를 재시작해야 하기 때문에 매우 비효율적


## 스프링 탄생 배경

스프링 프레임워크는 이러한 EJB의 복잡성과 무거움이라는 문제를 해결하기 위해 등장했다. 

> 스프링(Spring Framework) 은 자바 기반 애플리케이션을 더 가볍고 유연하게 개발할 수 있도록, 객체 관리(IoC/DI)와 공통 기능 분리(AOP) 를 핵심으로 제공하는 경량 애플리케이션 프레임워크이다.

탄생 배경 및 목표
- 배경 : EJB 가 제공하는 강력한 기술은 필요하지만, EJB 자체가 복잡하고 무거워서 효율성이 떨어지는 상황
- 목표 : EJB 장점을 유지하면서 더 가볍고 더 유연하고 더 생산적으로 개발 가능한 환경을 만드는 것
