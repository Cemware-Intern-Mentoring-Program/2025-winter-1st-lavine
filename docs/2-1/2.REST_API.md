## REST API , RESTful 설계

### REST
> - Representational State Transfer의 약자
> - 웹/HTTP에서의 해석: URI로 리소스를 식별하고, HTTP 메서드/헤더/상태코드/캐시 등 표준을 일관되게 사용해 조작한다.

### RESTful 설계
REST의 제약을 따르는 설계를 말한다.

REST에서 가장 중요한 규칙은 다음의 2가지이다.
- **URI로 정보의 자원을 표현(명사형 & 소문자 복수형)**
- **자원에 대한 행위는 HTTP Method(GET, POST, PUT, DELETE..)로 표현**

이에 대해 세부 규칙으로 다음과 같은 내용이 있다.
- 슬래시 구분자`/`는 계층 관계를 나타내는데 사용
- URI 마지막 문자로 `/`를 포함하지 않는다.
- 하이픈`-`은 URI 가독성을 높이는데 사용
- 밑줄 `_`은 URI에 사용하지 않는다.
- URI 경로에는 소문자 사용
- 파일확장자는 포함하지 않는다.

RESTful 제약 6가지가 있다.
- 클라이언트-서버 구조 : 클라이언트는 유저와 관련된 처리를, 서버는 REST API를 제공함으로써 각각의 역활이 확실하게 구분되고 일괄적인 인터페이스로 분리되어 작동할 수 있게 한다
- 무상태(Stateless) :  서버에서 어떤 작업을 하기 위해 상태정보를 기억할 필요가 없고 들어온 요청에 대해 처리만 해주면 되기 때문에 구현이 쉽고 단순해진다.
- 캐시 가능(Cacheable): 캐시 사용을 통해 응답시간이 빨라지고 REST Server 트랜잭션이 발생하지 않기 때문에 전체 응답시간, 성능, 서버의 자원 이용률을 향상 시킬 수 있다.
- 균일한 인터페이스(Uniform Interface): Uniform Interface는 Http 표준에만 따른다면 모든 플랫폼에서 사용이 가능하며, URI로 지정한 리소스에 대한 조작을 가능하게 하는 아키텍쳐 스타일을 말한다
- 자체 표현 구조 : JSON을 이용한 메시지 포멧을 이용하여 직관적으로 이해할 수 있고 REST API 메시지만으로 그 요청이 어떤 행위를 하는지 알 수 있다.
- 계층화(Layered System) : 클라이언트와 서버가 분리되어 있기 때문에 중간에 프록시 서버, 암호화 계층 등 중간매체를 사용할 수 있어 자유도가 높다

실무적으로 해석해보면 HTTP 규약을 지키고 리소스 지향적으로 모델링하며, 표준 메커니즘(메서드/상태코드/헤더/캐시/콘텐츠 협상)을 일관되게 활용하는 것.

HTTP 메서드 및 상태코드는 다음과 같이 정리할 수 있다.
- GET: 조회
- POST: 생성/서브리소스 추가/서버측 처리 트리거
- PUT: 전체 교체
- PATCH: 부분 수정
- DELETE: 삭제
- 응답 상태코드
    - 200 OK,
    - 201 Created(+Location 헤더)
    - 204 No Content
    - 400/401/403/404/409/412/415/422/429/500 등

> 항상 따르지는 못하지만 Restful하기 위해 노력한다고 생각하면 된다.
