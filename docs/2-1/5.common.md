> - @ModelAttribute와 @RequestBody는 어떨 때 이용하는 걸까요? 
> - @PathVariable @RequestParam은 무슨 차이일까요? 어떻게 해야 RESTful한 API를 만들 수 있을까요? 
> - 명세서는 어떻게 만들어야 할까요? 
> - Postman과 Swagger는 어떤 장단점이 있을까요? 또, Springdoc과 Springfox는 무슨 차이일까요? 
> - DTO는 어떻게 설계하는 것이 좋을까요?



### @ModelAttribute vs @RequestBody

`@ModelAttribute`는 요청 파라미터(QueryString, Form-data)를 객체에 바인딩할 때 사용한다.

`@RequestBody`는 HTTP Body(JSON/XML)를 읽어 객체로 역직렬화할 때 사용한다.

브라우저 Form 기반이면 @ModelAttribute, REST API JSON 기반이면 @RequestBody가 자연스럽다.


### @PathVariable vs @RequestParam
`@PathVariable`은 리소스 식별자(ex: /users/{id})에 사용하고,

`@RequestParam`은 검색 조건/옵션 파라미터(ex: ?page=1&sort=asc)에 사용한다.

RESTful하게 만들려면 리소스를 Path로, 조건은 Query로 설계하는 것이 좋다.



### API 명세서는 어떻게 만들어야 할까?

엔드포인트, HTTP 메서드, 요청/응답 스키마, 예시값, 상태 코드를 반드시 포함한다.


### Postman vs Swagger

Swagger는 API 설계 초기 단계에 스펙을 정의하고 문서화하면서 클라이언트/서버 구조를 정립하는데 사용한다.

Postman은 백엔드가 구현된 이후 실제 API를 호출하고 테스트 및 시나리오를 검증하는 등의 목적으로 사용한다.

두 도구를 보완적으로 함께 사용한다고 볼 수 있다.

### Springdoc vs Springfox
둘 모두 Spring을 이용해 API 문서 생성을 자동화하는 데 도움을 주는 라이브러리

- Springdoc: Spring Boot 3 / OpenAPI 3 공식 지원에 가까워 최신 스택에 적합
- Springfox: 업데이트 지연되고 Spring Boot 최신 버전과 호환성 문제가 있다.


### DTO 설계는 어떻게 해야할까?

> DTO(Data Transfer Object)란 계층간 데이터 교환을 하기 위해 사용되는 객체로 Getter/Setter만 가진 클래스

- dto는 간단해야한다. 즉, 로직 없이 데이터만을 담고 있어야한다.
- 빌더 패턴을 사용하여 유연한 DTO 객체 생성 가능
- 불변성을 지녀야한다. 한번 세팅하고 나면 프로퍼티에 변화가 있으면 안된다.
- 네이밍 컨벤션 일관되게 적용
- null 사용 피하기 :  null은 예상 못한 동작이나 에러를 발생시키기 때문에 -1이나 기본 값을 사용
- DTO를 사용하기 전에 DTO에 삽입될 값들이 유효한지 검증하는 유효성 체크를 해야한다.
